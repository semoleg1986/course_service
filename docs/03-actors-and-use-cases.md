# Акторы И Use Cases

## Акторы

- `Admin`: платформенное управление и модерация
- `Teacher`: создание/изменение собственного контента и оценивание
- `Parent`: оплата доступа ребенка и контроль учебного прогресса
- `Student`: прохождение уроков, сдача заданий/квизов
- `Attribution Service`: валидация referral token, расчет скидок, учет конверсий по каналам
- `Integration Service`: потребители событий (аналитика/уведомления)

## Ключевые Сценарии

### Поток Контента Teacher/Admin
1. создать курс в `draft`
2. добавить модули и уроки
3. добавить контент и цели уроков
4. определить prerequisites и расписание
5. добавить задания и quiz checkpoints
6. опубликовать курс

### Поток Доставки Для Student
1. родитель создает запрос доступа ребенка к курсу
2. referral token/канал валидируется и прикрепляется
3. админ вручную подтверждает оплату
4. админ одобряет доступ
5. ученик зачисляется на опубликованный курс
6. открывает доступный урок по правилам schedule/prerequisite
7. выполняет учебные активности
8. отправляет attempt по заданию/квизу
9. получает оценку и обратную связь

### Поток Родителя
1. родитель создает запрос доступа к курсу ребенка
2. родитель видит статус доступа (`requested | paid | approved | rejected`) и скидку
3. родитель читает прогресс ребенка по курсу
4. родитель читает список завершенных курсов ребенка

### Поток Управления Доступом
1. админ видит запросы доступа
2. админ переводит запрос в `paid`
3. админ одобряет или отклоняет доступ
4. `approved` разрешает enrollment, `rejected` блокирует enrollment

### Поток Управления Контентом
1. архивировать/разархивировать курс
2. передавать ownership преподавателя (контролируемо)
3. анализировать прогресс, завершения и метрики сабмитов

## Правила И Ограничения

- зачисление возможно только для `published` курса и `AccessGrant=approved`
- архивный курс неизменяем для delivery-операций
- доступ к уроку/модулю учитывает prerequisites и schedule
- оценивание допустимо только для отправленных попыток
